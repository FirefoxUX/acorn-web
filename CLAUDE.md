## Project Overview

Acorn Web extracts Firefox's `moz-*` web components and transforms them into standalone web components usable outside Firefox. It consists of a Rust transformation tool (`mozcomp`) and a Storybook frontend for viewing extracted components.

## Commands

### Rust (transformation tool)

```bash
cargo build                    # Build the tool
cargo build --release          # Build optimized binary
cargo test                     # Run all tests
cargo test test_name           # Run a specific test
cargo fmt                      # Format Rust code
cargo clippy                   # Lint Rust code

# Run the transformation (requires local Firefox source checkout)
cargo run -- /path/to/firefox ./dist ./config.toml
```

### Storybook (component viewer)

```bash
npm install                    # Install JS dependencies
npm run storybook              # Dev server on port 6006
npm run build-storybook        # Build static Storybook
```

## Architecture

### Transformation Pipeline (`src/`)

The Rust binary reads a Firefox source tree and outputs transformed web components to `dist/`. The pipeline runs in order:

1. **JAR Resolver** (`utils/jar_resolver.rs`) — Parses Firefox `jar.mn` files to map `chrome://` and `resource://` URLs to filesystem paths.
2. **Path Finder** (`utils/path_finder.rs`) — Resolves import paths (chrome://, resource://, relative) to absolute filesystem paths using the JAR resolver.
3. **Component & stylesheet discovery** (`pipeline/discovery.rs`) — Globs for `moz-*` components and global CSS using patterns from `config.toml`.
4. **Dependency graph** (`dependency_graph.rs`) — Builds a directed graph (`petgraph`) of all files and their imports. File types: `JsComponent`, `JsFile`, `CssFile`, `OpaqueFile`. Target locations: `Component`, `CssGlobal`, `Asset`, `Dependency`, `Omit`.
5. **Dependency extraction** (`dependencies/js.rs`, `dependencies/css.rs`) — Uses `oxc` (JS) and `lightningcss` (CSS) to parse import statements from source files.
6. **Transformation** (`transform/`) — Rewrites imports and URLs in JS/CSS files so they work outside Firefox:
   - `js_transform/url_transform.rs` — Rewrites import URLs
   - `js_transform/css_inline_transform.rs` — Inlines CSS into JS components
   - `js_transform/icon_template_import.rs` — Transforms icon template imports
   - `js_transform/import_css_transform.rs` — Transforms CSS imports in JS
   - `css_transform/import_replacer.rs` — Rewrites CSS `@import` paths
   - `css_transform/url_replacer.rs` — Rewrites `url()` references in CSS
   - `css_transform/icon_property_transform.rs` — Transforms `-moz-context-properties`/`context-fill` icon patterns to cross-browser `mask-image` approach
7. **Code generation** (`pipeline/codegen.rs`) — Generates `fluent-setup.mjs`, `acorn-icon.mjs`, and index files.
8. **File writer** (`pipeline/writer.rs`) — Transforms and writes all files to output directory.
9. **Docs** (`pipeline/docs.rs`) — Converts `.stories.md` to `.stories.mdx`.

### Key Rust Patterns

- **lightningcss** (v1.0.0-alpha.67): CSS nesting requires `ParserFlags::NESTING` (import from `stylesheet` module). When overriding `visit_rule()`, you **must** call `rule.visit_children(self)` or nested URL visiting breaks.
- **Icon transform pipeline**: SVGs get `context-fill` → `fill="black"` rewrite. CSS `-moz-context-properties` + `background-image: url(*.svg)` becomes `mask-image` with `-webkit-` prefix. JS `<img>` in html`` templates becomes `<acorn-icon>`. This uses string-based CSS post-processing, not lightningcss visitors, because `-moz-context-properties` is a custom property.
- **JAR resolver**: Firefox `jar.mn` files use `#ifdef`/`#ifndef` preprocessor directives (all conditions must be in the ifdef config) and can include `.inc.mn` files recursively via `#include`. Chrome registrations (`%` lines) are shared across all jar.mn files.
- **Error handling**: Unified `Error` enum in `src/errors.rs` with module-internal error types.

### Storybook (`.storybook/`)

- Reads stories from `dist/components/**/*.stories.mjs` (generated by the Rust tool)
- Framework: `@storybook/web-components-vite` (Storybook 9)
- Custom addons:
  - `addon-fluent/` — Mozilla Fluent localization with pseudo-localization support
  - `addon-component-status/` — Tracks component extraction status

### Configuration (`config.toml`)

Specifies which Firefox files to process: JAR manifest paths, mozbuild paths, component glob patterns, global stylesheet patterns, and docs paths. All paths are relative to the Firefox source root.

#### `[fluent_fallbacks]` — Manual Fluent fallback expressions

Some Fluent messages use variables (e.g., `{ NUMBER($rating, ...) }`) that can't be automatically converted to JS. For these, `config.toml` has an optional `[fluent_fallbacks]` table mapping l10n-ids to JS template expressions that are injected as `title=` attributes. When the build encounters `data-l10n-args` without a matching entry in this table, it emits a warning. All other Fluent fallback patterns are handled automatically from the FTL files.

### Output (`dist/`, gitignored)

The `package.json` exports components, styles, and assets from `dist/` as ES modules with `sideEffects: false`. Output directories: `components/`, `styles/`, `assets/`, `dependencies/`, `locales/`, `docs/`.
